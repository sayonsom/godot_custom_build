# app_spec.txt -- Godot 4.6.2-rc2 Fork Build Specification
# Target: Claude Code autonomous execution
# Deliverable: godot-lib.aar + .so files with AccessKit Android TalkBack + ThorVG Lottie

================================================================================
PROJECT OVERVIEW
================================================================================

Fork Godot Engine 4.6.2-rc2 to add:
  A) Native Lottie animation playback via ThorVG (already bundled, Lottie loader disabled)
  B) AccessKit Android adapter for TalkBack screen reader (full: nav order, live regions, custom actions, focus)

Final output: Android .aar and .so files for arm64-v8a, armeabi-v7a, x86_64.
Must work in BOTH standalone Godot Android apps AND embedded GodotFragment mode.

================================================================================
EXECUTION ORDER -- CRITICAL
================================================================================

Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4

Do NOT start Phase 2 until Phase 1 compiles and runs on Android.
Phase 1 (Lottie) validates the cross-compilation pipeline.
Phase 2 (AccessKit) is the high-risk work.

--- Step 1.4: Implement LottieAnimation node ---

    Create modules/lottie/lottie_animation.h:

    Class LottieAnimation : public Node2D
    Properties:
      - String lottie_file_path      // path to .json or .lottie file
      - float speed = 1.0
      - bool loop = true
      - bool autoplay = true
      - int current_frame             // read/write
      - int total_frames              // read-only
      - float duration                // read-only, seconds
      - Vector2 animation_size        // read-only, native Lottie canvas size

    Methods:
      - void play()
      - void pause()
      - void stop()
      - void seek(int frame)
      - bool is_playing() const

    Signals:
      - animation_finished()
      - animation_looped()

    Internal members:
      - tvg::SwCanvas* canvas = nullptr
      - tvg::Animation* animation = nullptr
      - Ref<ImageTexture> texture
      - PackedByteArray pixel_buffer
      - bool playing = false
      - float elapsed_time = 0.0

    Key implementation details:

    _ready():
      - Initialize tvg::Initializer::init(0)  // 0 = auto thread count
      - Create tvg::SwCanvas
      - Load lottie file via tvg::Animation
      - Get total_frames and duration from animation
      - Allocate pixel_buffer (width * height * 4 bytes, ARGB32)
      - Set canvas target to pixel_buffer
      - If autoplay, call play()

    _process(delta):
      - If not playing, return
      - elapsed_time += delta * speed
      - Calculate target_frame from elapsed_time and duration
      - Handle looping / finished
      - If frame changed:
        - animation->frame(target_frame)
        - canvas->clear()
        - canvas->update()
        - canvas->draw()
        - Convert ARGB32 → RGBA8 in pixel_buffer (swap channels)
        - Create Image from pixel_buffer
        - Update ImageTexture
        - queue_redraw()

    _draw():
      - If texture is valid:
        - draw_texture(texture, Point2())

    set_lottie_file_path(path):
      - Load the file
      - Reinitialize animation
      - Reset elapsed_time
      - queue_redraw()

--- Step 2.1: Build accesskit-c for Android ---

--- Step 2.2: Modify Godot build system for Android AccessKit ---

    File: platform/android/detect.py

    Find where env["accesskit"] is set. On desktop platforms it defaults to "yes".
    For Android it's probably missing or set to "no". Change to:
      env["accesskit"] = "yes"

    Or add a new SCons option:
      opts.Add(BoolVariable("accesskit", "Enable AccessKit accessibility", True))

    File: platform/android/SCsub

    Add linking for AccessKit static lib:
    ```python
    if env["accesskit"]:
        env.Prepend(CPPDEFINES=["ACCESSKIT_ENABLED"])
        abi = env["android_arch"]  # arm64, arm32, x86_64
        abi_dir = {"arm64": "arm64-v8a", "arm32": "armeabi-v7a", "x86_64": "x86_64"}[abi]
        env.Prepend(LIBS=["accesskit"])
        env.Prepend(LIBPATH=["#thirdparty/accesskit/android/" + abi_dir])
        env.Prepend(CPPPATH=["#thirdparty/accesskit/include"])
    ```

--- Step 2.3: Wire AccessKit into DisplayServerAndroid ---

    Study the existing desktop implementations first:
    - platform/windows/display_server_windows.cpp → search for ACCESSKIT_ENABLED

    File: platform/android/display_server_android.h

    - AccessibilityDriverAccessKit *accessibility_driver = nullptr;

    File: platform/android/display_server_android.cpp

    In DisplayServerAndroid constructor or create():
    - Initialize the AccessKit Android adapter
    - The accesskit-c Android adapter needs:
      a) A JNI env pointer (available via JNI_OnLoad or ThreadAndroid::get_env())
      b) A reference to the Activity or View that hosts the Godot surface
      c) A callback for when the adapter needs a tree update
    - Call accessibility_driver = memnew(AccessibilityDriverAccessKit)
    - Call accessibility_driver->window_create(MAIN_WINDOW_ID, android_native_handle)

    In has_feature():
    - case FEATURE_ACCESSIBILITY_SCREEN_READER:
        return (accessibility_driver != nullptr);

    The exact accesskit-c Android API functions will be visible in the accesskit.h header
    after building. They likely follow the pattern:
      accesskit_android_adapter_new(...)
      accesskit_android_adapter_update(...)

--- Step 2.4: Create Java-side accessibility bridge ---

    New file: platform/android/java/lib/src/org/godotengine/godot/accessibility/GodotAccessibilityDelegate.java

    ```java
    package org.godotengine.godot.accessibility;

    import android.view.View;
    import android.view.accessibility.AccessibilityEvent;
    import android.view.accessibility.AccessibilityNodeInfo;
    import android.view.accessibility.AccessibilityNodeProvider;

    /**
     * Bridges AccessKit's JNI-exposed accessibility tree to Android's
     * AccessibilityNodeProvider framework.
     *
     * AccessKit (Rust) → accesskit-c (C) → JNI → this class → Android TalkBack
     */
    public class GodotAccessibilityDelegate extends View.AccessibilityDelegate {
        // Native methods -- implemented in accesskit-c's JNI layer
        // These are called BY AccessKit when it needs to provide info to Android
        // and called FROM Android when TalkBack requests info

        // The exact JNI method signatures depend on accesskit_android's Java API.
        // Check the accesskit_android crate's Java companion class.
        // It likely provides a class like AccessKitNodeProvider that we delegate to.
    }
    ```

    IMPORTANT: The accesskit_android Rust crate already contains Java-side code
    (it uses the jni crate). We need to understand exactly what Java classes it
    expects to exist and wire them into Godot's Java layer.

    Check the accesskit_android crate source:
      https://github.com/AccessKit/accesskit/tree/main/platforms/android

    It likely provides:
    - A native method that creates the adapter
    - A Java class that implements AccessibilityNodeProvider
    - JNI callbacks for when TalkBack queries for node info

    Our job is to:
    1. Load the native library (libaccesskit.so or linked into libgodot_android.so)
    2. Create the adapter when GodotView is created
    3. Set GodotView's AccessibilityDelegate to use the AccessKit provider
    4. Ensure tree updates from DisplayServerAndroid reach the adapter

    File: platform/android/java/lib/src/org/godotengine/godot/GodotView.kt

    In init or constructor:
    ```kotlin
    // Wire accessibility
    if (accessibilityEnabled) {
        setAccessibilityDelegate(GodotAccessibilityDelegate(this))
    }
    ```

--- Step 2.5: Handle embedded mode ---

    In embedded mode, GodotFragment hosts GodotView inside the host app's view hierarchy.

    Key requirements:
    - GodotView.getAccessibilityNodeProvider() must return our AccessKit-backed provider
    - The provider must scope its virtual node IDs to not conflict with the host app
    - When TalkBack focus moves from host app views to GodotView, the transition must be smooth
    - When TalkBack focus is inside GodotView, navigation should follow AccessKit's tree order
    - AccessibilityEvents from Godot must be dispatched through GodotView, not the Activity

    Test specifically:
    - Host app has its own focusable views above/below GodotFragment
    - Swiping through TalkBack should enter and exit the Godot view cleanly
    - Live region announcements from Godot should not disrupt host app announcements

--- Step 2.6: Implement full TalkBack features ---

    For each feature, trace how it works on desktop and replicate for Android:

    NAVIGATION ORDER:
    - AccessKit tree children are ordered. This order becomes TalkBack swipe order.
    - Godot's Control.set_focus_next() and focus_neighbor properties should map to tree order.
    - Verify by creating a scene with multiple buttons and checking swipe order.

    LIVE REGIONS:
    - When a Control's text changes and it has accessibility_live = "polite" or "assertive":
      AccessKit sets LiveSetting on the node, the adapter sends TYPE_WINDOW_CONTENT_CHANGED.
    - TalkBack will announce the new text without moving focus.
    - Test with a Label that updates its text dynamically.

    CUSTOM ACTIONS:
    - Godot's accessibility_add_custom_action() adds a CustomAction to the AccessKit node.
    - On Android, this becomes an additional action in TalkBack's actions menu.
    - Test by adding a custom action to a Button and activating it via TalkBack.

    FOCUS MANAGEMENT:
    - When a Control grabs focus, AccessKit sets IsFocused.
    - The adapter sends ACTION_ACCESSIBILITY_FOCUS to TalkBack.
    - When TalkBack user explicitly focuses a node, the adapter sends Focus action back to Godot.
    - Test by programmatically grabbing focus and verifying TalkBack follows.

    TEXT INPUT:
    - LineEdit and TextEdit expose TextValue and TextSelection via AccessKit.
    - On Android, this maps to editable text AccessibilityNodeInfo.
    - TalkBack should announce characters as typed.
    - IME (software keyboard) should work with AccessKit's text input handling.
    - Test with a LineEdit: type text with TalkBack active, verify character echo.

    SCROLL:
    - ScrollContainer exposes ScrollPosition and handles ScrollTo action.
    - On Android, AccessibilityNodeInfo.setScrollable(true).
    - TalkBack can send scroll forward/backward actions.
    - Test with a VBoxContainer in a ScrollContainer with many buttons.

================================================================================
KNOWN GOTCHAS
================================================================================

1. accesskit-c's C header may not yet declare Android functions.
   You may need to add them to the header manually and regenerate:
     accesskit_android_adapter_new()
     accesskit_android_adapter_update_if_active()
     accesskit_android_adapter_free()
   Look at how windows/macos/unix adapter functions are declared in accesskit.h.

2. The accesskit_android Rust crate depends on the `jni` crate which needs
   $JAVA_HOME and Android JNI headers during compilation. If cargo-ndk fails,
   set these environment variables explicitly.

3. Godot's Android build uses both ndk-build style (for .so) and Gradle (for .aar).
   The .so is compiled by SCons directly using the NDK's Clang. Make sure the
   libaccesskit.a static lib was compiled with the SAME NDK version.

4. ThorVG's Lottie expressions feature uses a built-in JavaScript interpreter.
   This adds ~200KB+ and is a security concern. Disable it unless explicitly needed.
   Check ThorVG's build options for something like:
     -Dexpressions=false  (Meson) or a preprocessor define

5. In embedded mode, GodotView is a GLSurfaceView subclass. Android's accessibility
   framework treats SurfaceView/GLSurfaceView as opaque by default -- TalkBack will
   announce "Godot view" or similar. The AccessibilityNodeProvider override is what
   makes TalkBack see the individual nodes INSIDE the GL surface.

6. Godot's control focus system and AccessKit's focus concept are slightly different.
   Godot has keyboard focus (which Control processes input). AccessKit has accessibility
   focus (which screen reader highlights). Map Godot keyboard focus to AccessKit focus,
   but also handle the case where TalkBack moves a11y focus independently.

7. When building for arm32 (armeabi-v7a), ThorVG's NEON SIMD optimizations may not
   be available on all devices. ThorVG should gracefully fall back to scalar code.
   Verify this doesn't crash on older 32-bit devices.

================================================================================
SUCCESS CRITERIA
================================================================================

The build is DONE when:

1. `godot-lib-4.6.2-custom.aar` exists and contains .so for all 3 ABIs
2. A standalone Android app using the custom Godot:
   - Plays a Lottie animation via LottieAnimation node
   - All Control nodes are fully accessible via TalkBack
   - Navigation order, live regions, custom actions, focus, text input all work
3. An embedded Android app with GodotFragment:
   - Same Lottie + accessibility functionality as standalone
   - Host app's native views remain accessible
   - Focus transitions between host and Godot views work
4. Desktop builds (Linux/Windows) have no regressions:
   - Existing AccessKit screen reader support still works
   - LottieAnimation node works in the editor
5. Binary size overhead is < 500KB per ABI compared to vanilla Godot
